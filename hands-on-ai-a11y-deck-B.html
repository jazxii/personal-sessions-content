Why this works

Accessibility testing is made for LLMs

A lot of a11y work is reading code, looking up WCAG criteria, and writing structured reports. Three things modern AI is genuinely good at.

READING CODE
Spot missing labels, wrong roles, contrast gaps
LLMs trained on millions of HTML examples recognize a11y violations the same way they recognize syntax errors.
LOOKING UP WCAG
Criteria numbers, names, techniques
All 88 WCAG 2.2 A+AA criteria are in the training data. The LLM doesn't need to "remember" — it pattern-matches.
WRITING REPORTS
Defects, fixes, test cases
Structured text generation with consistent format is what LLMs literally exist to do.
The catch: the LLM will sometimes be confidently wrong. Your job is to use it for speed — and verify everything before it leaves your hands. We'll cover verification at the end.

Foundation · The prompt pattern

Every good a11y prompt has five parts

Learn this pattern once. Every prompt in this session is a variation on it.

1 · ROLE
Tell it who to be

"You are a senior accessibility engineer with deep WCAG 2.2 expertise." Frames the entire response.

2 · CONTEXT
Give it the WCAG criteria it should care about

Pasting relevant WCAG criteria text into the prompt = simulated RAG. Grounds the answer in real standards.

3 · TASK
State what you want done

"Audit this HTML and identify violations." One clear ask. Avoid mixing multiple tasks in one prompt.

4 · FORMAT
Specify the output structure

"For each issue: WCAG criterion, element, user impact, fix, severity." Structured output is parseable output.

5 · CONSTRAINTS
Tell it what NOT to do

"Don't give generic advice. Prefer semantic HTML over ARIA. Cite the exact WCAG criterion number."

EXAMPLE COPY-READY [ROLE] You are a senior accessibility engineer with deep WCAG 2.2 expertise. [CONTEXT] Most likely criteria for forms: - 1.3.1 Info and Relationships - 1.4.3 Contrast (Minimum) - 2.4.6 Headings and Labels - 3.3.2 Labels or Instructions - 4.1.2 Name, Role, Value [TASK] Audit the HTML below for WCAG 2.2 Level A and AA violations. [FORMAT] For each issue produce: - WCAG criterion (number + name) - Element (CSS selector or HTML) - User impact - Recommended fix with code - Severity: Critical/Major/Minor [CONSTRAINTS] Prefer semantic HTML over ARIA. Don't give generic advice. Group issues by WCAG criterion. --- HTML --- <paste HTML here>

Foundation · Grounding the AI

Simulated RAG: paste the standard into the prompt

You don't need a knowledge base server. You can ground the AI by pasting the relevant WCAG criteria directly into your prompt. This dramatically reduces hallucinations.

WHY IT WORKS
★
LLMs trust their context window over their training data. Paste the criterion text, and the model will cite it accurately.
★
It fixes the wrong-WCAG-number problem. Without context, LLMs sometimes invent criterion numbers. With context, they don't.
★
It works on free tiers. No paid tools, no plugins, no APIs. Just paste.
★
It scales to your project. Add your client's design system tokens, custom rules, brand glossary — all as pasted context.
Pro tip: Save a "WCAG context block" in your notes app. Paste it at the top of every a11y prompt. Two minutes once, every prompt better forever.
SAVE THIS BLOCK REFERENCE — WCAG 2.2 A & AA (core): Perceivable 1.1.1 Non-text Content — Provide text alternatives. 1.3.1 Info and Relationships — Structure programmatically determinable. 1.3.5 Identify Input Purpose — autocomplete attributes. 1.4.3 Contrast (Minimum) — 4.5:1 text, 3:1 large. 1.4.10 Reflow — No 2D scroll at 320px. 1.4.11 Non-text Contrast — 3:1 for UI components. Operable 2.1.1 Keyboard — All functionality keyboard-accessible. 2.1.2 No Keyboard Trap — Focus can leave any element. 2.4.3 Focus Order — Logical sequence. 2.4.6 Headings and Labels — Describe topic/purpose. 2.4.7 Focus Visible — Visible focus indicator. 2.5.3 Label in Name — Visible label matches accessible name. Understandable 3.2.1 On Focus — No context change on focus. 3.2.2 On Input — No context change on input. 3.3.1 Error Identification — Errors described in text. 3.3.2 Labels or Instructions — Provided for input. Robust 4.1.2 Name, Role, Value — All UI components. 4.1.3 Status Messages — Programmatically determinable.

★

LAB 2 · GENERATE FIXES

Get the AI to write the fix, not just describe it

The difference between "this needs an aria-label" and "here's the corrected component, ready to paste." One ticket gets fixed faster than the other.

PROMPT · ARIA/HTML FIX GENERATOR COPY ME You are a senior frontend accessibility engineer. I will paste an HTML/JSX snippet with accessibility issues. Your job: TASK: 1. Identify what's wrong (cite the WCAG criterion) 2. Output the fixed HTML/JSX — full snippet, ready to paste 3. Explain why your fix works (1-2 sentences max) 4. If you added ARIA, justify why native HTML wasn't sufficient CONSTRAINTS — First Rule of ARIA applies: - Prefer semantic HTML over ARIA every time - No ARIA > bad ARIA - Don't introduce new accessibility issues - Preserve the visual design — change semantics only - Keep the same framework (HTML stays HTML, JSX stays JSX) - Don't add features that weren't there before OUTPUT FORMAT: ## Issue [what's broken + WCAG ref] ## Fixed Code ```html <corrected snippet> ``` ## Why this works [1-2 sentences] ## ARIA justification (if used) [why native HTML wasn't enough] --- BROKEN SNIPPET --- <paste broken HTML/JSX here>

Important · Things AI gets wrong

Where AI trips you up — and how to catch it

Every technique in this deck is also a way to confidently produce wrong work. Know these pitfalls before your SBA.

1. Made-up WCAG criterion numbers.
Without context, LLMs occasionally cite "WCAG 2.5.7" or other non-existent criteria.
Fix: Always paste the WCAG context block. Always verify criterion numbers against W3C's official list.
2. Over-reliance on ARIA.
The AI loves to sprinkle aria-label, aria-labelledby, role="..." everywhere — even where native HTML would work.
Fix: Always include the "First Rule of ARIA — prefer semantic HTML" constraint in your prompt.
3. False positives in audits.
AI flags color contrast violations from inline styles when the actual computed style is fine. It can't see the rendered DOM.
Fix: For contrast checks, verify with browser DevTools or axe DevTools before filing. Don't trust HTML-only inference.
4. Generic, non-specific recommendations.
"Add alt text" — but to which image? "Improve contrast" — to what ratio? Useless tickets.
Fix: Add the constraint "be specific to this snippet, no generic advice" to every audit prompt.
5. Broken Playwright code.
AI sometimes uses older axe-core APIs or missing imports. Doesn't run on first try.
Fix: Always run what the AI generates. If it errors, paste the error back into the chat and say "fix this." Two rounds usually does it.
6. Confident hallucinated fixes.
"Add role='presentation' to the table." (Don't.) AI sometimes confidently suggests fixes that worsen accessibility.
Fix: If a suggested fix surprises you, look it up before applying. Trust your training over the model when they conflict.

The four-step verify habit

Before you file anything the AI wrote

Build this habit on day one. It's the difference between a strong a11y engineer and someone who outsourced their judgment.

1
Verify WCAG numbers

Open w3.org/WAI/WCAG22/quickref and confirm every criterion number cited. Takes 30 seconds, catches the most common AI mistake.

2
Test the fix in the browser

For ARIA/HTML fixes — actually paste them into a browser. Verify with axe DevTools or NVDA before sending the suggestion to a dev.

3
Run the test script

For Playwright code — npx playwright test it. If it errors, iterate with the AI. Don't ship untested code.

4
Sanity-check against your training

If the AI suggests something that contradicts what you learned in onboarding — pause. Look it up. Your knowledge is the override.

Time math: Without AI = 60 min per audit. With AI (no verification) = 8 min, low quality. With AI + 4-step verification = 20 min, high quality. That's still 3x faster than manual, and the work you ship is trustworthy.

SBA prep

What you should be able to demonstrate

If your SBA gives you a broken page and a chat tab, here's what a confident demonstration looks like.

DEMONSTRATE
Open Claude/ChatGPT/Gemini and run a full audit prompt with WCAG context grounding

The assessor wants to see the WCAG context block, the role+task+format pattern, and the structured output. Show them you understand the *anatomy*, not just the words.

DEMONSTRATE
Chain audit output → defect writer prompt for one violation

Take one issue from the audit and produce a complete JIRA-format ticket. Mention the Priority field and the {{BUILD_LINK}} placeholder — that signals you understand the team's standards.

DEMONSTRATE
Generate a fix for one violation with First Rule of ARIA applied

Pick one violation. Get the AI to produce a fixed snippet. Explain why semantic HTML beats ARIA. Bonus: paste the fix into a browser and confirm it passes axe DevTools.

DEMONSTRATE
Generate and explain a Playwright + axe-core test script

Run the base prompt, walk through the generated code. If you can run it locally, even better. Mention what you'd add for a real project (CI/CD integration, custom rules, report storage).

DEMONSTRATE
Identify at least one thing the AI got wrong — and correct it

This is the most important moment. Showing critical thinking — "the AI suggested this, but it's wrong because..." — is what separates a passing demo from a strong one.
